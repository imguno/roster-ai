# Roster 철학

## 하나의 원칙: Organization as Code

Infrastructure as Code가 서버를 프로비저닝하듯, Roster는 조직을 프로비저닝합니다.

YAML로 desk, group, resource, 라우팅을 선언합니다.
Roster는 실제로 일하는 팀을 세웁니다 — 항상 실행 중이고, 항상 준비된 상태로.

---

## 당신은 CEO입니다

Roster는 소프트웨어 추상화가 아닌 현실 세계의 조직을 모델링합니다.

- **에이전트를 채용합니다** — 누구인지, 무엇을 아는지(skill), 무엇을 배웠는지(knowhow)를 정의합니다
- **desk를 배정합니다** — 에이전트를 executor(LLM, 스크립트, 사람, 원격)에 연결합니다
- **팀을 구성합니다** — 컨텍스트를 공유하고 협업하는 desk 그룹을 만듭니다
- **라우팅을 설정합니다** — 부서 간 업무 흐름처럼, 이벤트가 팀 사이를 흐릅니다
- **관찰하고 개입합니다** — 대시보드로 현황을 파악하고, 이벤트를 emit해 방향을 조율합니다

DAG, 폴링 간격, 메시지 큐는 신경 쓸 필요 없습니다.
그것은 Roster의 역할입니다.

---

## 모든 것은 이벤트입니다

커뮤니케이션 패턴은 딱 하나입니다: **이벤트**.

- 팀은 작업이 완료되면 이벤트를 emit합니다
- 다른 팀은 그 이벤트를 구독합니다
- Resource는 세상이 변할 때 이벤트를 emit합니다 (파일 수정, PR 오픈, 타이머 발화)
- 라우팅 규칙이 생산자와 소비자를 연결합니다

```
design-team emits design.approved
  → routing delivers to dev-team
  → dev-team implements, emits code.ready
  → routing delivers to qa-team
  → qa-team tests, emits qa.passed
  → routing delivers to deploy-team
```

팀 간 통신과 팀 내 통신을 위한 별도 문법이 없습니다.
출력 텍스트에 특별한 마커가 없습니다. 컴포넌트 간 커스텀 프로토콜이 없습니다.
이벤트가 들어오고, 이벤트가 나갑니다. 그게 전부입니다.

---

## Resource는 공유 상태입니다

Resource는 관찰 가능한 레이어입니다. 조직을 외부 세계와 서로 연결합니다.

- **task-board** resource는 디렉터리입니다. 파일을 쓰면 대상 팀이 트리거됩니다.
  디렉터리를 열면 — 지금까지 배정된 모든 task가 보입니다.
- **codebase** resource는 git을 감쌉니다. Desk가 커밋하고, 파일을 읽고, 스크립트를 실행할 수 있습니다.
- **slack** resource는 메시지를 전달합니다. LLM이 필요 없습니다 — 스크립트면 충분합니다.

Resource에는 **actions**(쓰기, 커밋, 알림)과 **watcher**(파일 변경, 폴링)가 있습니다.
Watcher는 이벤트를 emit하고, action은 exec 프로토콜을 통해 desk가 호출합니다.

핵심 인사이트: resource는 커뮤니케이션을 **관찰 가능하고 지속적으로** 만듭니다.
이벤트는 일시적입니다. Resource 상태는 영속적입니다. 팀은 둘 다를 통해 소통합니다.

---

## Group은 팀이지, 파이프라인이 아닙니다

Group은 컨텍스트를 공유하는 desk 팀입니다. 단계들의 순서가 아닙니다.

```yaml
kind: group
name: dev-team
lead:
  desk: implementer
groups:
  - impl-squad      # 중첩 group: 구현
  - review-squad    # 중첩 group: 리뷰
```

Lead가 조율합니다: 먼저 계획하고, 마지막에 종합합니다. 중간에 멤버들이 작업합니다.
멤버들은 **group session**을 통해 서로의 결과물을 볼 수 있습니다 — 실행 내에서 쌓이는 공유 컨텍스트입니다.

**Conversation 모드** (`dispatch: conversation`): 멤버들이 여러 라운드에 걸쳐 서로 응답하며 실행됩니다.
고정된 파이프라인이 아니라 유기적인 토론입니다 — 각 멤버가 다른 멤버의 말을 발전시키거나, 반박하거나, 확장할 수 있습니다.

멤버는 추가할 것이 없을 때 **SKIP**할 수 있습니다. 이것은 실패가 아닙니다 — 자율적 판단입니다.
시간이 지나면서, 각 desk는 자신의 기여가 필요한 시점을 스스로 학습합니다.

---

## 판단이 필요한 곳엔 LLM, 그렇지 않은 곳엔 스크립트

모든 곳에 LLM이 필요한 건 아닙니다.

| LLM이 필요한 경우 (판단, 창의성) | 스크립트로 충분한 경우 (결정적 처리) |
|---|---|
| 디자인 리뷰 | 빌드 / 컴파일 |
| 코드 구현 | 테스트 수트 실행 |
| 콘텐츠 작성 | 파일 변환 |
| 우선순위 결정 | 알림 전달 |
| 데이터 분석 | 정기 백업 |

LLM 호출은 시간과 비용이 듭니다. 셸 스크립트는 즉각적이고 무료입니다.
상황에 맞는 도구를 사용하세요.

## Knowhow는 쌓입니다

Skill은 에이전트를 채용할 때 갖추는 것 — 미리 정의된 지식입니다.
Knowhow는 에이전트가 일하면서 배우는 것입니다.

```
skills/                 # 사람이 작성하고, git으로 버전 관리
  product-strategy.md
  code-review.md

knowhow/                # 작업에서 추출되어, 시간이 지나며 축적
  common-pitfalls.md
  customer-patterns.md
```

Desk의 결과물에 `## Knowhow` 섹션이 포함되어 있으면, hub가 이를 추출하여
knowhow 디렉터리에 저장합니다. 다음에 그 에이전트가 실행될 때,
학습된 패턴이 프롬프트에 포함됩니다.

Skill은 agent로 조합됩니다. Knowhow는 시스템이 학습하면서 축적됩니다.

---

## 중앙 통제보다 자율 거버넌스

Roster는 세세하게 관리되는 파이프라인보다 자율적인 팀을 지향합니다.

- **SKIP**: 추가할 것이 없는 desk는 그렇게 말하고 빠집니다.
  불필요한 LLM 호출 없이, 억지로 결과물을 만들지 않습니다.
- **Conversation**: Group의 멤버들은 순서대로 실행하는 게 아니라 토론합니다.
  두 명의 구현자가 디스패처를 기다리지 않고 자연스럽게 작업을 나눌 수 있습니다.

목표: 조직을 한 번 정의하면 스스로 돌아가게 합니다.
시스템이 당신 없이 작동할 수 없어서가 아니라, 방향을 잡고 싶을 때만 개입하면 됩니다.

---

## Artifact는 작업의 산출물입니다

모든 desk는 artifact를 만들어냅니다. 모든 artifact는 파일로 저장됩니다.

```
.roster/data/artifacts/
  design-team-designer-20260608-1000-a3b4c5d.md
  dev-team-implementer-20260608-1100-f6g7h8i.md
  qa-team-tester-20260608-1200-j1k2l3m.md
```

Artifact는 팀이 소통하는 방식입니다. 디자인 팀의 목업이 개발 팀의 스펙이 됩니다.
개발 팀의 코드가 QA 팀의 테스트 대상이 됩니다.
같은 artifact가 Slack으로, 대시보드로, 체인의 다음 팀으로 전달됩니다.

이를 통해 숨겨진 내부 상태가 **눈에 보이는 작업 산출물**로 대체됩니다.
artifacts 디렉터리를 열면 — 모든 결정, 모든 결과물, 모든 인계 과정이 보입니다.

---

## 설계 원칙

1. **Orchestration이 아닌 Organization** — 소프트웨어가 작업을 스케줄링하는 방식이 아니라, 팀이 일하는 방식을 모델링합니다
2. **하나의 커뮤니케이션 패턴** — 이벤트와 resource, 모든 곳에서, 모든 것에
3. **기본적으로 관찰 가능** — 파일을 열면 무슨 일이 있었는지 보이고, 대시보드를 열면 무엇이 실행 중인지 보입니다
4. **영속적인 상태** — 큐, 세션, knowhow는 지속됩니다; 크래시가 나도 복구 가능합니다
5. **최소한의 개입** — 팀이 스스로 운영됩니다; 사람은 운영이 아닌 방향 설정만 합니다
6. **적절한 도구** — 판단에는 LLM, 결정적 처리에는 스크립트
7. **조합 가능** — desk, group, skill, resource는 재사용 가능한 빌딩 블록입니다
8. **메시지보다 Artifact** — 작업 산출물은 이벤트 스트림에서 사라지지 않고 파일로 저장됩니다
